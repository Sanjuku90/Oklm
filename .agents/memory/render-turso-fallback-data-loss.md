---
name: Render deploy silently loses data on restart
description: App falls back to ephemeral local SQLite in production when remote DB env vars aren't configured on the host dashboard, causing data loss on every restart/redeploy.
---

Root cause pattern: an app is built to use a remote database (e.g. Turso/libSQL) when credentials are present, with a silent fallback to a local SQLite file otherwise (`USE_TURSO = bool(creds present)`, decided once at process startup). `render.yaml` (or equivalent IaC) lists the remote DB credentials as `sync: false`, which means the hosting platform will never auto-populate them — they must be pasted manually into the platform's dashboard as env vars for that specific deployed service. If a user forgets this step (or the values get lost/reset), the app boots successfully, works fine in every manual/browser test, but every write goes to a local file that lives only in the deploy's ephemeral container filesystem — wiped on every restart, redeploy, or scale event.

**Why:** This is dangerous specifically because it's invisible during normal testing — the app "works," registration/login/deposits all return success, and the bug only becomes apparent hours/days later when the service restarts (which happens routinely on most PaaS platforms, e.g. free-tier idling on Render) and all data disappears. It's easy to mistake for a totally different bug (e.g. "registration is broken") because the visible symptom is "I have to re-register."

**How to apply:** When a user reports "I lose all my data after a restart" / "I have to re-register every time" on a deployed (non-Replit) host, immediately check: (1) does the app have a dual local/remote persistence path gated by an env-var presence check, (2) does the IaC config mark those remote DB credentials as `sync: false` / manual-only, (3) is there any confirmation the values were actually pasted into the *host's* dashboard (not just available in the dev environment's secrets — dev and deployed prod environments have separate env var stores). Also add a loud, unmissable startup log/warning that detects "running on this host but falling back to non-persistent local storage" so this is diagnosable from platform logs immediately next time, rather than requiring code archaeology.
